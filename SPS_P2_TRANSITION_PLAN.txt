ARCHITECTURE RECAP (what stays the same)
----------------------------------------
The repo is a set of cooperating processes, usually each in its own tmux session
(start_servers.sh):

  daq_control.py        orchestrator: loops over sub_runs, asks hv_control to
                        ramp+monitor, asks dream_daq_control to take data
  hv_control.py         TCP server (port 1100): sets CAEN HV, ramp check,
                        monitor thread -> hv_monitor.csv per subrun
  dream_daq_control.py  TCP server (port 1101): builds .cfg from template,
                        drives Dream RunCtrl, copies fdfs to run_out_dir
  processor_watcher.py  watches runs/ for new fdfs: decode -> analyze_waveforms
                        (hit trees) -> combine_feus_hits  (mm_dream_reconstruction
                        C++ executables)
  qa_watcher.py         watches combined_hits_root/ and runs a python QA script
                        per subrun (currently nTof_x17/ntof_daq_analysis/detector_qa.py)
  pedestal_watcher.py / backup_watcher.py / trigger_logger.py   auxiliaries
  flask_app/            web GUI: start/stop run, watcher toggles, HV live plot,
                        run/subrun progress

All of this is generic. What is nTOF-specific and must change:
  (a) run_config_beam.py   detectors, HV crate + channel map, IPs, paths, trigger
  (b) processor_config.py  executable paths + data paths
  (c) qa_config.py         QA repo path + QA entry script (nTOF -> P2)
  (d) flask_app/app.py     BASE_DIR and BASE_DATA_DIR constants
  (e) hardware-facing code has no simulation mode -> needed for local testing
