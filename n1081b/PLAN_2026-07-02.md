## 1. Firmware upgrade of 242 (web GUI) — walk-through

I'll walk you through this live. The SDK has **no firmware method**, so this is
GUI/touchscreen only.

**Before you start:** you still need the **firmware package** for `2025.3.27.0`
(or newer) from CAEN's login-walled portal — it's *not* among what you downloaded
(you have the manual, datasheet, CERN talk, SDK zip). Grab it from
<https://www.caen.it/download/> → search "N1081B" → firmware. Drop it somewhere the
browser you're using can reach.

Steps:

1. **Back up 242's config to your computer first.** Firmware flashing can wipe the
   active config. Options: General Settings → Configuration → select
   `homogeneous_wire.json` → **Download** (saves the .json locally). We also already
   hold `snapshots/dump.json` and the on-board `backup_pre_homog.json`.
2. Confirm you're on 242 (Search Device beep).
3. Note current versions: General Settings → **Version Info** (should read SW
   `2023.12.4.0`, plus ZYNQ/FPGA). Compare against a good board (e.g. 240 =
   `2025.3.27.0`) so we know the target.
4. General Settings → **Firmware Upgrade** tab → **Search file** → pick the firmware
   file → the **Upgrade** button enables → click **Upgrade**.
5. **Do not interrupt power / network** during the flash. The board reboots itself.
6. After reboot: **Version Info** should now show `2025.3.27.0` (match the others).
7. Re-apply our config if the upgrade reset it: Configuration → load
   `homogeneous_wire.json` (or re-run `homogenize.py` — it's idempotent). Verify with
   `dump_module_info.py` + `summarize_dump.py`.

**Risks:** 242 lives in a NIM crate — make sure power is stable. If the flash fails
mid-way the board can need a recovery/factory procedure (touchscreen). Have the
manual (`docs/WEB_UM8139_x1081B_rev1.pdf`, §"General Settings … firmware upgrade",
p.82) open.

## 2. Password — nothing to disable

Findings:

- The **web GUI has no login/password at all** — zero mentions of password/login in
  the 90-page manual; access over Ethernet is open. So there's nothing to "disable"
  there; it's already open on the DAQ net.
- The **only** password is the **WebSocket API** `login` command. Firmware **rejects
  an empty password** (`change_password("")` → `Result:false`, `login("")` fails), so
  it can't be blanked. The change↔login round-trip is otherwise reversible (tested).
- **Decision:** keep the default `password`. Our tooling auto-logs-in, so it's already
  transparent. Nothing to do.

(If you ever want *some* client to skip it, the only lever is a shorter/agreed
password via `change_password`, not a blank — but we don't need it.)

## 3. Trigger-recording test (time-tag logger)

Goal: prove `trigger_logger.py` records real triggers from a unit and lock the CSV
schema. Start with **one** unit, then fan out.

### What the datasheet/manual tell us (important)

- Time Tagging: from a **common T0**, a 64-bit counter ticks in **10 ns** steps; when
  an enabled input commutates, `(CH, Timestamp in ns)` is captured via FIFO. **The
  board already gives ns** — no tick math needed.
- **Only inputs CH 1,2,4,5 are time-taggable** (manual, 1-based). In the SDK's
  0-based `configure_time_tagging(en0..en5)` that's **lemo 0,1,3,4**. CH 3 / CH 6
  (SDK lemo 2 / lemo 5) are reserved (veto / external T0) — enabling them for tagging
  won't work.
- **T0** is internal (set a T0 frequency) or **external on CH 6 (SDK lemo 5)**. A
  common external T0 fanned out to CH 6 of every unit is the clean way to align
  timestamps **across units** (this is the "common start/sync pulse" we flagged).

### Steps

1. **Deploy the code to the server** (so it runs natively there): commit `n1081b/`
   + trigger files, push, `git pull` on `ssh daq`. SDK already installed in the
   server venv.
2. **Point the logger at a real unit.** Edit `trigger_config.py`:
   - `UNITS = [{name:'n1081b_240', ip:'192.168.10.240', password:'password',
     section:'A', channels:[0], input_standard:'NIM', impedance:50}]`  (channel 0 =
     CH 1, a valid tag input).
   - `output_dir` → a real test dir, e.g. `/mnt/data/x17/tests/triggers/`.
   - Regenerate: `.venv/bin/python trigger_config.py`.
3. **Provide triggers + a T0.** Easiest self-test without external pulsers: set one
   section to **pulse_generator** and cable its output → the time-tag input (and a T0
   into CH 6). Or feed known NIM pulses from the existing trigger fan-out. (Decide
   which physical signals we tap.)
4. **Run:** `ssh daq` → in the repo venv →
   `python trigger_logger.py config/trigger_config.json`. Send a few pulses.
5. **Inspect** `n1081b_240_triggers.csv` — capture one real packet's fields. Confirm
   the `(CH, Timestamp_ns)` schema and **pin the CSV columns** in
   `trigger_logger._packet_columns/_packet_values` (right now they're inferred from
   the first packet). Update `docs/n1081b_trigger_logger.md`.
6. **Robustness:** verify clean Ctrl-C shutdown and auto-reconnect (pull/replug).
7. **Fan out** to 240/241/242/243 (each its own thread/file). Discuss whether to add
   244 (still in use).

### Open questions to resolve during the test

- Which real signals we route into the units for a physics-meaningful test.
